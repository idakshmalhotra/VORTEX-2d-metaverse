# WebRTC Video/Audio Implementation

## Overview
Added peer-to-peer video/audio communication using WebRTC, automatically triggered by proximity detection in the metaverse app.

## Architecture

### Server Side (apps/ws)

#### 1. WebSocket Signaling (user.ts)
- Added WebRTC signaling message handlers:
  - `OFFER` - Forward WebRTC offer to target user
  - `ANSWER` - Forward WebRTC answer to target user
  - `ICE_CANDIDATE` - Forward ICE candidates to target user
  - `CALL_END` - Forward call end signal to target user
- Added `forwardWebRTCMessage()` method to route signaling messages between users
- Added `getUsers()` method in SpaceManager (server.ts) to retrieve users in a space

#### 2. Message Format
All WebRTC signaling messages include:
- `type`: Message type (OFFER, ANSWER, ICE_CANDIDATE, CALL_END)
- `from`: Sender sessionId
- `to`: Target sessionId
- `offer`/`answer`/`candidate`: WebRTC data (depending on type)

### Client Side (apps/frontend)

#### 1. WebRTCManager (game/service/WebRTCManager.ts)
Modular peer connection manager that handles:
- Media access (camera/microphone) via `getUserMedia()`
- RTCPeerConnection lifecycle management
- WebRTC signaling (offer/answer/ICE candidate exchange)
- Proximity-based automatic connections
- Connection cleanup on disconnect

Key methods:
- `startMedia()` - Request camera/microphone access
- `initiateCall(remoteSessionId)` - Create offer and start connection
- `handleOffer(remoteSessionId, offer)` - Handle incoming offer, create answer
- `handleAnswer(remoteSessionId, answer)` - Handle incoming answer
- `handleIceCandidate(remoteSessionId, candidate)` - Add ICE candidates
- `closeConnection(remoteSessionId)` - Close specific peer connection
- `closeAllConnections()` - Cleanup all connections
- `toggleMic(enabled)` / `toggleCamera(enabled)` - Control media tracks

ICE Configuration:
```typescript
{
  iceServers: [
    { urls: "stun:stun.l.google.com:19302" }
  ]
}
```

#### 2. Network Integration (game/scenes/Network.ts)
- Initialize WebRTCManager after receiving sessionId
- Handle WebRTC signaling messages from WebSocket
- Emit PROXIMITY_ENTER/PROXIMITY_LEAVE events
- Close connections on PLAYER_LEFT or PROXIMITY_LEAVE
- Cleanup all connections on WebSocket disconnect

#### 3. Proximity Integration (game/scenes/MyPlayer.ts)
- Listen to PROXIMITY_ENTER event → initiate WebRTC call
- Listen to PROXIMITY_LEAVE event → close WebRTC connection
- Only initiate call if local media stream is available

#### 4. Event Bus (game/scenes/EventBus.ts)
Added new events:
- `PROXIMITY_ENTER` - Emitted when user enters proximity
- `PROXIMITY_LEAVE` - Emitted when user leaves proximity

#### 5. Video Overlay (components/WebRTCVideoOverlay.tsx)
React component that renders:
- Local video preview (corner, green border)
- Remote video feeds (overlay, blue border, tagged with sessionId)
- Control buttons (camera toggle, microphone toggle)
- Status display (connection count, camera/mic status)

#### 6. React Hook (hooks/useWebRTC.ts)
Custom hook to integrate WebRTCManager with React:
- Polls for WebRTCManager availability from window.gameNetwork
- Sets up callbacks for remote stream events
- Provides methods for media control
- Manages local state for streams and UI

#### 7. Game Page Integration (app/game/page.tsx)
- Added WebRTCVideoOverlay component
- Integrated useWebRTC hook
- Exposed network to window.gameNetwork for hook access

## Connection Flow

### On PROXIMITY_ENTER:
1. Server emits PROXIMITY_ENTER to both users
2. MyPlayer receives event via EventBus
3. If local stream is available, WebRTCManager.initiateCall() is called
4. RTCPeerConnection created, local tracks added
5. Offer created and sent via WebSocket signaling

### On receiving OFFER:
1. Network receives OFFER message from WebSocket
2. WebRTCManager.handleOffer() called
3. RTCPeerConnection created, local tracks added
4. Remote description set, answer created
5. Answer sent via WebSocket signaling

### On receiving ANSWER:
1. Network receives ANSWER message from WebSocket
2. WebRTCManager.handleAnswer() called
3. Remote description set on existing connection

### On ICE candidate:
1. ICE candidate generated by RTCPeerConnection
2. Sent via WebSocket signaling to peer
3. Peer adds candidate via handleIceCandidate()

### On PROXIMITY_LEAVE:
1. Server emits PROXIMITY_LEAVE to both users
2. MyPlayer receives event via EventBus
3. WebRTCManager.closeConnection() called
4. RTCPeerConnection closed, tracks stopped
5. CALL_END signal sent to peer

## Usage

1. User joins a room and enters the game
2. Click camera button in WebRTCVideoOverlay to enable camera/microphone
3. When users come within 100px proximity, video call starts automatically
4. When users move apart (>100px), video call ends automatically
5. Toggle camera/microphone using the control buttons

## Testing

To test 2-user video call:
1. Open app in two browser windows/tabs
2. Join the same room in both
3. Enable camera in both windows
4. Move avatars close together (<100px)
5. Video should connect automatically
6. Move avatars apart (>100px)
7. Video should disconnect automatically
